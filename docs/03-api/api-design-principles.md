# API Design Principles

## الهدف
توحيد تصميم واجهات برمجة التطبيقات (API) في EduCore لضمان الاتساق، وقابلية التوسع، وسهولة التكامل مع تطبيقات الويب والجوال والأنظمة الخارجية.

## المبادئ الأساسية
- REST-first architecture.
- JSON هو تنسيق البيانات القياسي.
- استخدام HTTPS فقط.
- جميع الموارد تستخدم UUID كمُعرّف.
- تصميم Stateless.

## قواعد التسمية
- استخدام أسماء الموارد بصيغة الجمع.
- أمثلة:
  - GET /api/v1/students
  - POST /api/v1/students
  - GET /api/v1/students/{id}

## الاستجابات
- استخدام أكواد HTTP القياسية.
- تغليف الاستجابة في هيكل موحد.

## الأخطاء
- إرجاع Error Code وMessage وTraceId.
- عدم كشف تفاصيل داخلية للنظام.

## الإصدارات
- استخدام Versioning عبر المسار: /api/v1/
- الحفاظ على التوافق مع الإصدارات السابقة قدر الإمكان.

## Rate Limiting
- تطبيق حد أقصى لعدد الطلبات لكل مستخدم أو مفتاح
API

في نافذة زمنية محددة
(مثال: 100 طلب/دقيقة للعمليات القرائية، 20 طلب/دقيقة للعمليات الكتابية الحساسة).

- إرجاع كود
429 Too Many Requests

مع
Header

باسم

Retry-After

عند تجاوز الحد.

- تطبيق حدود أكثر صرامة على
Endpoints

الحساسة تحديداً:

POST /api/v1/auth/login

و

POST /api/v1/finance/payments

لمنع محاولات التخمين أو إساءة الاستخدام المالي.

## Idempotency
- كل عملية
POST

على موارد مالية
(الدفعات تحديداً)

يجب أن تدعم
Header

اسمه

Idempotency-Key

يرسله العميل
(Client).

- لو نفس المفتاح اتكرر خلال 24 ساعة، النظام يرجع نفس الاستجابة الأصلية من غير ما ينفذ العملية تاني، عشان يمنع تسجيل دفعة مكررة بسبب إعادة إرسال الطلب
(Retry)

من الشبكة أو تطبيق الجوال.

- يُحتفظ بمفاتيح
Idempotency

لمدة 24 ساعة على الأقل في طبقة
Caching

مثل

Redis.

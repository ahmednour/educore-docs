# ADR-009: ZATCA E-Invoicing Strategy

## Status
Accepted

## Context
يخدم
EduCore

مراكز تعليمية متنوعة، بعضها مرتبط رسميًا بهيئة الزكاة والضريبة والجمارك
(ZATCA)

ومُلزَم بمنظومة الفوترة الإلكترونية
(فاتورة / Fatoora)،

وبعضها الآخر لا يعمل بشكل رسمي مع الدولة من الناحية الضريبية حاليًا. النظام يحتاج دعم الحالتين من غير ما يفرض على كل الفروع نفس المتطلبات، ومن غير ما تحتاج إعادة هيكلة لاحقًا لو فرع قرر يتسجل رسميًا بعدين.

## Decision
- تفعيل الفوترة الإلكترونية يكون على مستوى الفرع وليس على مستوى النظام كله، عبر عمود
branches.zatca_enabled.

- الفروع اللي
zatca_enabled = false

تصدر فواتير عادية بدون
QR Code

أو أي استدعاء لمنصة

ZATCA

الخارجية، وده الوضع الافتراضي.

- الفروع اللي
zatca_enabled = true

لازم يتوفر عندها

zakat_tax_number،

ويتفعّل توليد

zatca_uuid

و

zatca_qr_code

تلقائيًا مع كل فاتورة.

- التكامل الفعلي مع منصة
ZATCA

(الاعتماد

Compliance

والإرسال المستمر

Reporting)

يُنفَّذ في طبقة

Application

كخدمة منفصلة قابلة للتفعيل، مش منطق مدموج جوه

Controller

الفوترة العام.

- أي فرع يقرر التسجيل رسميًا بعدين، العملية تكون تفعيل إعداد فقط
(Toggle)

من غير أي تعديل على بنية قاعدة البيانات أو فقدان لبيانات الفواتير القديمة.

## Alternatives Considered
- إلزام كل الفروع بمنظومة
ZATCA

من اليوم الأول — تم الرفض لأنه يفرض تعقيد وتكلفة غير مبررة على فروع غير مسجلة رسميًا.

- بناء نظام فوترة منفصل تمامًا لكل حالة — تم الرفض لأنه يضاعف الصيانة ويكسر توحيد جدول
invoices

الواحد.

## Consequences
### الإيجابيات
- مرونة كاملة لكل فرع حسب وضعه الرسمي الفعلي.
- عدم وجود أي تكلفة تقنية أو تعقيد إضافي على الفروع غير المسجلة.
- جاهزية فورية لأي فرع يقرر التسجيل لاحقًا من غير Migration جديد.

### السلبيات
- الحاجة لاختبار السيناريوهين
(مفعّل / غير مفعّل)

بشكل منفصل في كل دورة اختبار للوحدة المالية.

- منطق شرطي إضافي في طبقة
Application

عند توليد أي فاتورة.
